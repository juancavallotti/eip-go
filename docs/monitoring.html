Monitoring & logs

Once an integration is deployed, two things tell you how it's doing: its logs, aggregated centrally and searchable across every deployment, and its status — pods coming up, running, or failing — streamed live into the editor. This page covers both, and how they get there.

Centralized logs

Every log line, in one place.

Anything a flow logs — via the log block or the runtime itself — is shipped off the pod and collected centrally, so you read one stream for an integration no matter how many replicas it runs. The editor's logs view queries that history.

What you can do

  • Filter by app/deployment, level (e.g. just ERROR + WARN), a time range, and a free-text search on the message.
  • Tail live — the view polls the newest page and appends rows, sticking to the bottom until you scroll up.
  • Page back through history with keyset pagination (newest first; default 200 rows per page).
  • Share a view — filters mirror to the URL, so a filtered query is a link.

What a log record carries

deploymentIdWhich deployment emitted it.
appName / appVersionIntegration name and the version tag running.
ts / levelTimestamp and slog level.
messageThe rendered log message.
attrsAny extra structured fields, kept as JSON.
How it gets there. Each pod ships structured logs to NATS (internal.logs, fire-and-forget); a log-aggregator service consumes them as a competing consumer and writes them to Postgres; the editor reads that table. The path and the schema are in platform architecture.
Live status

Deployment & pod health.

The orchestrator watches the cluster with a Kubernetes informer and recomputes a deployment's status whenever a pod changes. That status streams to the editor over server-sent events, so you watch a deploy go live in real time rather than refreshing.

  • Rollout — pods going from pending to ready as a deploy or version rollout proceeds, and the desired vs. ready replica count.
  • Failures — a pod that crash-loops or fails to schedule surfaces on the deployment, with the reason from Kubernetes.
  • Scale — change replicas and the count updates live as pods are added or removed.

Pair this with the logs view: status tells you a replica is unhealthy; the logs tell you why.

What's in scope today

Logs and status — not yet metrics.

Observability today is centralized logs plus live deployment status. A few things are deliberately not here yet, so you know where the edges are:

  • Flow events (started / completed / dropped / failed) stay on a process-local bus — they aren't shipped to a central trace store. Log what you need to see.
  • There's no Prometheus/metrics endpoint or queue-depth dashboard yet; backpressure is enforced at the flow level but not surfaced as a metric.
