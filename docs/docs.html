Octo documentation

Octo is a cloud-native, AI-native integration platform. You compose flows in a readable YAML language; a small, concurrent runtime carries them; and a Kubernetes-native platform deploys, exposes, and operates them. These docs are organized around that split — learn the runtime model first, then how to build, run, and deploy.

The mental model

Two layers: a runtime, and a platform.

Almost everything in Octo belongs to one of two layers. Keeping them separate is the fastest way to understand how the pieces fit.

⚙️

The runtime

A small Go engine that loads a YAML config and runs it. Connectors turn the outside world into messages; sources feed them into flows; flows run them through an ordered chain of blocks. It runs standalone — a single binary, no platform required.

☸️

The platform

A control plane (orchestrator + Postgres) and a visual editor that store integration definitions and deploy them as Kubernetes workloads — each integration its own pods, with env binding, secrets, scaling, and a public TLS endpoint.

A flow at a glance

What an integration looks like.

A config declares connectors and one or more flows. A flow binds a source and lists the blocks each message passes through. This one fires on a schedule and logs a line.

service:
  name: hello-world

connectors:
  - name: ticker
    type: cron
  - name: out
    type: logger

flows:
  - name: greet
    source:
      connector: ticker
      type: cron
      settings:
        schedule: "@every 5s"
    process:
      - type: log
        logger: out
        settings:
          message: '"hello world! the time is " + string(now)'
Try it. Save as hello.yaml and run octo -config hello.yaml, or call a single flow directly with octo invoke -config hello.yaml -flow greet. See Running standalone.
