Clustering & replicas

On the platform an integration can run as several replicas — pods of the same runtime image behind one Service. This page is the mental model for what that means: how work is balanced across replicas with queues, how shared state stays consistent under concurrency, how run-once work picks a single owner, and why one integration's state never bleeds into another's.

The unit of clustering

One deployment, many replicas.

When you deploy an integration you choose replicas. Each replica is a pod running the same definition; together they form one logical service. Everything clustered is scoped to the deployment — keyed by OCTO_DEPLOYMENT_ID — so the replicas of one integration share their queues, object store, and leader leases with each other and with nothing else on the cluster.

ConcernShared across a deployment's replicas via
Work distributionCompeting-consumer queues (NATS queue groups)
Shared stateThe object store, versioned (KV over HTTP to the orchestrator)
Run-once workLeader election (Kubernetes Lease objects)
Standalone collapses all of this. Run the same definition with the standalone build and there is exactly one process: queues are in-memory, the store is in-process, and that process is always the leader. The flow code is identical — only the backends differ. See platform architecture.
Load balancing

Spreading work across replicas with queues.

An http source is fronted by the Kubernetes Service, which round-robins requests across pods — fine for request/reply. For work that should be processed once, by exactly one replica, with backpressure, use a queue. A queue source makes every replica a competing consumer of the same subject: each message is delivered to one replica only, so adding pods adds throughput.

Consume — the queue source

Binds a flow to a subject as a competing consumer. listeners adds in-pod concurrency; with request/reply, the flow's result is sent back as the reply.

flows:
  - name: worker
    source:
      connector: bus
      type: queue
      settings:
        subject: work
        listeners: 4      # concurrent handlers in this pod
    process:
      - { type: log, settings: { message: '"handling " + body.id' } }

Produce — the queue-dispatch block

Sends the current message to a subject. Fire-and-forget by default; set awaitReply to do request/reply and fold the response back into the message.

- type: queue-dispatch
  settings:
    subject: '"work"'       # CEL → route or shard by content
    awaitReply: false
    # timeout: 10s          # when awaitReply: true
In a cluster the runtime scopes the subject to the deployment (octo.<deploymentId>.q.<subject>) over NATS; standalone, it's an in-process channel. Either way the flow YAML is the same. Full settings: the queue connector.
Concurrency

How the object store stays consistent.

When replicas share state, two of them can touch the same key at once. The object store handles this with optimistic concurrency: every entry carries a version, and a write only succeeds if the version it expects still matches.

  • Create vs. update — writing with expected version 0 creates the key; a positive expected version updates only if it hasn't changed since you read it.
  • Conflicts are loud, not silent — if another replica wrote first, the version no longer matches and the write returns a conflict rather than clobbering their value.
  • Read-modify-write is handled for you — the object-write block re-reads and retries on a conflict, so a counter incremented by three replicas lands at the right number.

There is no native TTL; the cache-scope block layers expiry on top by stamping a deadline into the value and treating expired entries as a miss. The block-level APIs (object-read, object-write, object-delete, cache-scope) and the namespaces they use are documented under State & clustering.

Run exactly once

Leader election for run-once work.

Some work must happen once per cluster, not once per replica — a scheduled tick, a single poller, a migration. Three replicas with a naive cron source would fire three times. Leader election prevents that: replicas campaign for a key and only the winner does the exclusive work.

Distributed cron

The cron source campaigns for a per-schedule key and only the leader emits a tick — so the schedule fires once cluster-wide no matter how many replicas run. You get this for free; nothing to configure.

Under the hood

In the cluster, leadership is a coordination/v1 Lease per key, scoped to the deployment (15s lease, 10s renew, 2s retry). If the leader pod dies, the lease lapses and another replica takes over. Standalone, the single process is always the leader.
