Build & run

Running standalone

The runtime is a single Go binary. You don't need the platform, a database, or Kubernetes to run a flow — point octo at a config and go. The same binary powers a local visual editor in Docker for designing flows against files on disk.

Quickstart

Running in under a minute.

1 · Clone & build

# requires Go 1.25+ and Task
git clone https://github.com/juancavallotti/octo
$ cd octo && task build

2 · Run a sample

$ task run:sample -- hello-world.yaml

A cron tick logs a greeting on a schedule.

3 · Check the version

$ ./bin/octo --version
# → octo 0.2.0 (built 2026-06-18T22:31:23Z)

4 · Develop with hot reload

$ ./bin/octo --config samples/http-orders.yaml --watch

Edit the YAML (or a .env), save, and the runtime rebuilds itself.

The octo CLI

Three ways to run a config.

Run mode — start sources and services

The default. Starts every connector and flow; sources begin emitting. --config accepts a file or a directory (every YAML in it is loaded). Ctrl-C unwinds the lifecycle cleanly.

$ octo --config samples/hello-world.yaml

Invoke mode — call one flow directly

Runs a single flow by name with JSON in and JSON out — no sources started, no ports bound. The cleanest way to test a flow. Sourceless flows are perfect for this.

$ octo invoke \
  --config samples/hello-invoke.yaml --flow greet \
  --data '{"name":"Ada"}'
# → {"greeting":"hello, Ada!"}

Watch mode — hot reload

Add --watch to any run. The runtime rebuilds on every config or .env change, preserving runtime services across reloads. Edit a flow, save, watch it take effect — no restart.

$ octo --config flows/ --watch
Tip. A flow with no source: gets an implicit source and becomes callable by name — ideal for invoke and for flow-ref sub-flows.
Configuration

Environment and .env.

  • Environment variables override config defaults — e.g. HTTP_PORT=9090 octo --config ....
  • A .env file (in the working dir, or at $OCTO_ENV_FILE) is sourced automatically.
  • LOG_LEVEL controls verbosity (debug / info / warn / error).
  • Names referenced as ${VAR} in settings must be declared in the config's env: block.
Visual editor

The public Docker image.

Prefer to design flows visually without the full platform? A published, self-contained image — juancavallotti/octo on Docker Hub, rebuilt on every release — bundles the editor, the octo runtime, and an MCP server, with no orchestrator, auth, or database. Flows are read and written as *.yaml under /work; mount a host directory there.

$ docker run -p 3000:3000 -v "$PWD:/work" juancavallotti/octo
  • Editor at http://localhost:3000; the bundled octo binary powers the RUN feature in-container.
  • OCTO_FS_DIR — the flow store (the image sets it to /work).
  • OCTO_DOCS_URL — optional; surfaces a docs link in the MCP authoring prompt.
Built-in MCP server. The image also exposes an unauthenticated /mcp endpoint (http://localhost:3000/mcp), so a local AI assistant can read the runtime catalogue, draft and validate a definition, run it, and read the logs — all against your files. See Authoring with MCP.
