Build & run

Building integrations

An integration is a YAML config: you declare the connectors it needs and the flows that process messages. This page covers the config schema, the difference between leaf and composite blocks, the enterprise integration patterns they map to, and — when the built-ins aren't enough — how to add your own connector or block in Go.

The config schema

Anatomy of a definition.

A definition has five top-level keys. Only flows is strictly required; the rest appear as your integration grows.

service:                      # identity + metadata
  name: orders
  environment: prod

env:                          # declared env vars, usable as ${NAME} and in CEL `env`
  - name: API_KEY
    required: true

connectors:                   # instances of connectors this integration uses
  - name: api
    type: http
    settings: { port: 8080 }
  - name: out
    type: logger

processors:                   # optional: reusable, named block definitions
  - name: audit-log
    type: log
    logger: out
    settings: { message: '"audit: " + eventID' }

flows:                        # one or more root flows
  - name: orders-api
    workers: 8                # pool size (default 1 = FIFO)
    buffer: 128               # source channel depth (default 64)
    pool: 16                  # shared pool for concurrent composites (default 8)
    source:
      connector: api
      type: http
      settings: { path: /orders/{id} }
    process:
      - type: log
        logger: out
        settings: { message: '"got order " + body.id' }
    error:                    # root-only recovery chain
      - type: log
        logger: out
        settings: { message: '"failed: " + vars.error.message' }
Env interpolation. Settings accept ${ENV_VAR} from the declared env: block. The same names are available to CEL expressions via the env variable. Referencing an undeclared name is an error at build time, not in production.
Composing flows

Leaf blocks and composite blocks.

Leaf blocks do one thing to a message — log it, set its payload, call an HTTP endpoint, run a query. Composite blocks embed sub-flows through typed slots, which is how you get branching, loops, parallelism, and error boundaries. Both are listed in the connectors & blocks reference.

Every block below ships with the runtime. Connector-backed blocks name their connector with a connector: (or logger:) setting; the full settings for each live in the reference.

Extending the runtime

Writing a custom connector.

When you need to own a resource the built-ins don't cover, implement the Connector interface and self-register it. If it should feed messages into a flow, also provide a source.

package myconnector

import (
    "context"
    "github.com/juancavallotti/octo/core"
    "github.com/juancavallotti/octo/types"
)

func init() {
    core.MustRegisterConnector("my-api", func() core.Connector { return &Connector{} })
}

type Connector struct{ /* owns a client, pool, listener, ... */ }

func (c *Connector) Start(ctx context.Context, cfg types.ConnectorConfig) error {
    // acquire resources
    return nil
}
func (c *Connector) Stop(ctx context.Context) error {
    // release resources
    return nil
}

// Optional: provide a source so this connector can feed a flow.
func (c *Connector) NewSource(cfg types.SourceConfig, out chan<- *types.Message) (core.MessageSource, error) {
    return &source{out: out}, nil
}
Extending the runtime

Writing a custom block.

A leaf block is a MessageProcessor: one method that transforms a message. Decode its settings from the config, register it by type, and reference it from a flow.

package myblocks

import (
    "context"
    "github.com/juancavallotti/octo/core"
    "github.com/juancavallotti/octo/types"
)

func init() {
    core.MustRegisterBlock("my-transform", newMyTransform)
}

type settings struct {
    Multiplier float64 `json:"multiplier"`
}
type myTransform struct{ multiplier float64 }

func newMyTransform(raw types.Settings, deps core.BlockDeps) (core.MessageProcessor, error) {
    var cfg settings
    if err := raw.Decode(&cfg); err != nil {
        return nil, err
    }
    return &myTransform{multiplier: cfg.Multiplier}, nil
}

func (m *myTransform) Process(ctx context.Context, msg *types.Message) (*types.Message, error) {
    // mutate msg.Body / msg.Variables, then:
    //   return msg, nil   → pass through
    //   return nil, nil   → drop (filtered)
    //   return nil, err   → fail (error path)
    return msg, nil
}

Reference either from a flow once the package is imported into your build:

connectors:
  - name: my-api
    type: my-api
    settings: { baseUrl: https://api.example.com, apiKey: ${API_KEY} }

flows:
  - name: process-data
    source: { connector: my-api, type: webhook, settings: { path: /data } }
    process:
      - type: my-transform
        settings: { multiplier: 2.5 }
Composite blocks (new typed slots) are a deeper extension point handled by the flow builder rather than the block registry — reach for a leaf block or compose existing composites first.
