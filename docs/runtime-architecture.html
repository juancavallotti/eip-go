Concepts & runtime

Runtime architecture

The runtime is a small Go engine that loads a YAML config and runs it. A connector turns the outside world into messages; a source feeds them to a flow; the flow runs them through an ordered chain of blocks. That's the whole model — and it composes all the way down.

The processing model

From an external event to a processed message.

Five concepts cover everything the runtime does: message, connector, source, flow, and block. A flow is itself a message processor, so flows nest inside blocks and the model repeats at every level.

The pieces

Message, connector, source, block, flow.

Message

The unit of work. Every message carries a stable eventID, an optional caller-supplied correlationID, a mutable vars map for per-message state, and a JSON body (objects decode to maps, numbers to floats). Messages are JSON-only by design — no binary payloads — and clone safely (deep copy) when a flow needs to process branches concurrently.

Connector

Owns a piece of the outside world — a socket, a client, a pool, a file — acquiring it on Start and releasing it on Stop. Connectors do two things: some provide a source that feeds messages into a flow (cron, http, queue); some provide a block a flow calls by name (a logger, a database, an HTTP client, an LLM). They self-register at startup, so adding one is a matter of importing a package.

Source

A flow's entry point, created and owned by a connector. It reacts to an external event — an HTTP request, a cron tick, a queue message — by building a Message and sending it on a channel the runtime owns. A flow without a declared source: gets an implicit source and becomes callable by name (ideal for invoke and flow-ref sub-flows).

Block

A configured, named stage in a flow. A block's processor transforms one message and returns one of three outcomes:

  • pass through — returns the (possibly modified) message;
  • drop — returns (nil, nil); the message is filtered and ends as dropped;
  • fail — returns an error; the message takes the error path and ends as failed.

Leaf blocks are built from the block registry (log, set-payload, rest, sql, …). Composite blocks embed sub-flows. See the connectors & blocks reference.

Flow

An ordered list of blocks — and itself a message processor, which is what makes composition recursive. A root flow binds a source and may set workers (pool size, default 8; set 1 for FIFO), buffer (channel depth, default 64), pool (shared pool for concurrent composites, default 8), and an error recovery chain. Sub-flows nested in composites declare only their process chain.

Concurrency

Worker pools and backpressure.

Each root flow runs on a dedicated pool of workers reading one bounded channel. Backpressure is built in: when workers fall behind, the channel fills and the source blocks. A second, shared pool powers concurrent composites like fork. Set workers: 1 for strict FIFO; raise it for throughput (messages may then complete out of order).

Observability. The flow-event bus stays inside the process. What leaves a pod is your structured logs: in a cluster deployment the runtime ships them to the platform's central log aggregator, searchable across every integration; standalone, they go to stderr.
Lifecycle

One process, a clear start/stop discipline.

A service owns the order of operations: connectors acquire resources first, flows build and start, then sources begin to emit. On shutdown everything unwinds in strict reverse — sources stop, channels close, workers drain, connectors release.

Composition

Composite blocks — flows inside blocks.

A composite block embeds sub-flows through explicit, typed slots. handle-errors is the single-threaded half (an error boundary with recovery); fork is the concurrent half (scatter, then join). Both keep the one-in / one-out seam intact, which is why they nest cleanly. Other composites — if, switch, foreach, cache-scope, and the AI composites — follow the same shape.
