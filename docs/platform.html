Platform

Platform architecture

The runtime is one process. The platform around it is cloud-native: a control plane — the editor, the orchestrator, Postgres, NATS, and a log aggregator — and a data plane where every integration you deploy becomes its own Kubernetes Deployment of the same runtime image, wired to clustered backends for state, queues, and logs.

The shape

A control plane and a data plane, joined by NATS.

You design flows in the editor; the orchestrator stores them in Postgres and applies Kubernetes manifests; the data plane runs the pods. NATS is the message backbone between the two — it carries inter-replica queues, ships logs to the aggregator, and streams deployment status back to the editor.

Components

What runs in the control plane.

Editor

A Next.js app — the visual canvas where you build flows, manage folders and integrations, configure deployments, bind secrets, and read logs. It talks to the orchestrator through a server-side proxy (a BFF), so the browser never calls the orchestrator directly and the orchestrator stays internal and unauthenticated. Sign-in is OIDC SSO. The same editor also ships as a standalone image backed by local files instead of the orchestrator.

Orchestrator

The Go control-plane API. It owns integration definitions and deployment records in Postgres, is the only component that talks to the Kubernetes API, and serves the runtime's clustered KV store over HTTP (/deployments/<id>/kv/<namespace>/<key>, with optimistic version headers; secret namespaces encrypted at rest). It also publishes deployment-status snapshots to NATS.

NATS

The message backbone. It is not used by the runtime in standalone mode at all — it appears only on the platform, where it serves three jobs:

  • Queues — deployment-scoped subjects (octo.<deploymentId>.q.<subject>) with queue groups, so the replicas of one integration are competing consumers and a message is handled by exactly one pod. Native request–reply is supported.
  • Log shipping — every runtime pod publishes structured log records to the shared internal.logs subject, fire-and-forget.
  • Deployment events — the orchestrator publishes status snapshots on internal.deployments.<integrationId>, which the editor relays to the browser as SSE.

Log aggregator

A small standalone service that subscribes to internal.logs as a competing consumer (queue group octo-logs), parses each JSON record — pulling out deploymentId, app name/version, timestamp, level, and message, with the rest kept as JSONB — and writes it to the logs table in Postgres. The editor's logs view queries that table (filter by deployment, level, time, and text; keyset pagination; tail by polling the newest page).

One runtime, two builds

How the runtime wires to standalone or the cluster.

The runtime's backing services — the object/KV store, secret store, leader election, queues, and the log sink — sit behind interfaces with two implementations. Which one compiles in is chosen by a build tag; which one activates at startup is confirmed by the RUNTIME_SERVICES_MODULE environment variable. The two builds share all the same flow code — only the backends differ.

ServiceStandalone build (default)Cluster build (-tags k8s)
Object / KV storeIn-process, in-memory, versionedHTTP to the orchestrator (Postgres-backed)
Secret storeIn-process namespacesEncrypted at rest by the orchestrator
Leader electionAlways leader (single process)Kubernetes Lease objects
QueuesIn-process channelsNATS queue groups, deployment-scoped, request–reply
LogsstderrShipped to NATS → log aggregator → Postgres
How the build is selected. The default go build (the octo CLI and the standalone image) compiles the in-process providers. Building with -tags k8s swaps in the clustered providers and the Kubernetes/NATS clients; this is the image the orchestrator deploys. At deploy time the orchestrator injects RUNTIME_SERVICES_MODULE=k8s plus OCTO_DEPLOYMENT_ID, ORCHESTRATOR_URL, NATS_URL, and the pod's name/namespace, so the same image self-configures per deployment.
Events & observability

Two event streams: flow events and platform events.

The runtime emits a flow event for every message — started then one of completed/dropped/failed — on a process-local bus that connectors use to correlate request/response. These stay inside the pod. What leaves the pod and reaches you is:

  • Structured logs — anything a flow logs ships to the central aggregator and shows up, searchable, in the editor's logs view across every deployment.
  • Deployment status — pod readiness, failures, and replica counts stream from the orchestrator (via a Kubernetes informer) over NATS and SSE, so the editor shows pods going live in real time.
